Why This Project

市场入口 · 项目背景与商机判断

老龄化社会加速，独居老人的情感陪伴与健康监护是正在爆发的细分市场。大多数切入这一赛道的创业公司具备硬件研发能力，但缺乏自研大模型的团队规模和资金——这正是云端 AI 赋能方案的典型入口：我们提供「大脑」，客户提供「身体」。

客户计划开发一款挂式 AI 陪伴设备，支持 Wi-Fi / 4G，初期语音交互为核心，后续可扩展视觉模态。Q1 完成百台量级样机，Q2 小批量，年度目标十万台，节奏紧，对接入速度要求极高。

行业趋势
GUI 时代的边界在哪里
屏幕与 APP 对老年人构成天然门槛。语言交互界面让设备使用门槛归零——不需要用户学习，戴上就会用。这是市场壁垒，不只是功能差异。
商机判断
为什么这个客户值得优先推进
年度十万台量级意味着十万份 License；客户使用竞品云，存在迁移商机。拿下一个 ISV，等于间接覆盖其全部下游终端用户——这是乘数效应，不是单点项目。
Problem Framing

客户问题定义 · 表层诉求与深层驱动

客户表面上在找「一套 AI 方案」，但真实需求分属三个层次：技术可行性 / 体验达标 / 商业模式安全。识别这三层，是方案讲解切入点不同的原因。

01
无自研 AI 能力，需快速集成成熟方案
硬件团队不具备大模型研发能力，Q1 出样机节奏极紧，需要完整 SDK 文档、快速接入、完善技术支持保障。
对应切入：SDK 源码开放 + 最快 1 天接入 + 支持低成本嵌入式芯片（RTOS）
硬性需求
02
交互延迟是体验生死线，老年用户容忍度极低
客户明确提出 ≤1.5s 的延迟要求，且当场追问两次。一旦感知卡顿，老年用户会认为「设备坏了」，直接影响产品口碑。
对应切入：全链路端到端音频延时低至 1.3s；VAD 精准检测说话结束，最大化降低主观等待感
硬性需求
03
主动关怀是核心差异化卖点，不是可选功能
客户将「基于历史记忆主动发起关怀对话」作为向投资人讲故事的核心卖点。这是融资叙事的基础，而非功能偏好。
对应切入：主动任务引擎（规划中）+ 长短期记忆 + 情绪感知；同时需进行期望值管理
高度关注
04
成本结构必须可预测，服务融资计划书
按量付费模型无法精确测算硬件 BOM，需要一次性授权模式支撑商业计划书中的单台成本模型。
关键话术：License = 可预测成本 = 融资故事里稳定的利润结构锚点
商业诉求
05
数据自主：对话历史需超 7 天，支持本地管理
计划自建数据管理平台，需支持超 7 天存储并可 API 导出——背后是对数据主权与未来 AI 个性化训练数据的战略布局。
对应切入：对话历史导出 API；存储容量上限为信息缺口，需内部产研确认
商业诉求
Solution Overview

方案总览 · 架构逻辑与交互式架构图

核心架构逻辑是「轻端重云」：把所有 AI 计算放在云端，终端设备只负责采集输入与播放输出两件事。低成本嵌入式芯片无需升级，也能支撑完整的多模态 AI 体验——硬件 BOM 成本保持可控，同时云端能力随时升级推送，无需召回设备。

业务价值
架构决策
为什么推轻端重云，而不是端侧部署
客户芯片无法跑本地大模型，但更关键的是：云端方案支持远程升级。今天集成的 SDK，未来新能力可直接推送，硬件不需要召回升级——这是产品长期竞争力，不只是技术选型。
性能决策
1.3s 不是参数，是用户感知「活人」的生死线
开源拼接方案往往 3-4 秒延迟。1.3s 和 3s 的差距不是量变，是「伙伴感」和「复读机感」的质变。体验差 1 秒，老年用户的产品信任差一倍。
性能指标永远要翻译成用户感受，而不是留在参数表里。
生态决策
30+ 预置应用卖的不是功能，是上市时间
大模型谁都有，但天气、地图、音乐的商务对接需要巨大工作量。「开箱即用」帮客户省掉至少半年的内容生态建设周期——Time-to-Market 才是这个功能的真实价值。
商业决策
License 是融资叙事里成本确定性的锚点
同一个定价，两种说法结果完全不同。「按量付费」让投资人看到变动成本；一次性 License 让单台 BOM 成本固定，可以精确写进商业计划书的成本模型。
SA 的价值之一：把技术特性翻译成商业决策语言。

Highlight Moment

现场高光时刻 · 主动关怀的场景化诠释

现场贡献 · 可复述的核心故事
从「技术功能」到「老人的朋友」——一个场景类比打开了客户的想象空间

在讲解「主动任务 / 情绪感知」模块时，客户方对这一功能的实际效果仍有疑问。我主动介入，用具体的老年用户场景做了一个即兴类比：

「您设想一下，老人家之前跟设备说过最近胸口不舒服。两天后早上，设备可以主动说：今天心率看起来正常，感觉好些了吗？或者说：今天是不是也出去晨练了？——这样的设备对老人家来说更像一个真正的朋友，而不是等着被问的机器。」

这个表达让客户立刻抓住了差异化价值——从「被动响应」到「主动陪伴」。这也是他们产品在老年硬件市场能建立护城河的核心卖点，是融资叙事的情感支撑点。

Retrospective

三个真实挑战与复盘

01
数据存储问题当场答不上来
发生了什么

客户追问对话数据存储容量上限和保存时长，我当时对这块产品细节不熟悉，无法给出准确答案，现场出现停顿。

处理方式与成长

请教同事后意识到：提前拉产研入会实时解答，比一个人独自覆盖所有知识盲区更专业。「不知道」不是失误，「没有机制兜底」才是——团队协作本身是 SA 能力的一部分。

02
POC 后续流程不熟悉，推进节奏不清晰
发生了什么

账号开通后，不清楚测试验收标准是什么、谁来跟进 SDK 集成、什么条件下进入商务谈判。节奏感缺失影响了主动推进的自信。

成长方向

SA 的完整能力不只是「懂方案」，还包括掌握售前全流程的节奏感。初访 → 账号开通 → POC 验收 → 商务谈判，每个节点的目标、推进方式、内部协调路径都需要系统积累。

★
核心判断：功能未上线，如何应对客户最看重的那个需求
两难困境

客户最看重的「主动关怀」在拜访时仍是规划功能，尚未落地。过度承诺导致 POC 失败信任崩塌；直接说没有则客户认为方案不成熟当场流失。这是售前最常见也最难处理的期望管理困境。

观察到的处理方式

坦诚说明规划上线时间 → 展示「定时提醒」作为当前可用的过渡方案 → 给出明确测试安排承诺。诚实 + 过渡方案 + 时间节点，三者缺一不可。这是我在这个项目里对 SA 角色理解最深的一处。

Core Learning

售前不是销售。销售可以夸大，但 SA 是客户技术决策链上信任的建立者——短期的诚实比长期的失信成本低得多。这个判断，不是从产品手册里学来的，是在真实客户的问题里踩出来的。
